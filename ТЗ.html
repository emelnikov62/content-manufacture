Концепция и техническое задание

Веб-приложение для управления соцсетями нескольких бизнес-направлений («Контент-завод»)

Версия: 1.3
Дата: 27 июня 2026
Статус: Концепция для передачи разработчику

История изменений: v1.1 — добавлены Facebook (Страницы) и X (Twitter). v1.2 — зафиксированы: AI-слой на kie.ai (текст + изображения + видео + аудио), платный провайдер трендов, ~2–4 аккаунта на направление; апрув-воронка исключена. v1.3 — бюджет $100–300/мес и контроль расходов (раздел 12), провайдер трендов = EnsembleData, генерация видео подтверждена в Фазе 2.

1. Краткое описание

Веб-приложение, в котором владелец нескольких бизнес-направлений и его небольшая команда управляют всеми социальными сетями (Instagram, TikTok, Telegram, Threads, Facebook, X/Twitter) из одного места. Аккаунты сгруппированы по бизнес-направлениям. Реализован пайплайн полного цикла работы с контентом — от поиска идей и AI-генерации до автопостинга и аналитики, — а также воронки авто-ответа по ключевому слову (лид-магниты).

Слой публикации и входящих сообщений строится на Postproxy (unified-провайдер), чтобы не реализовывать и не сопровождать интеграции с каждой соцсетью отдельно и не проходить долгие аудиты платформ.

Бизнес-цели

  1. Сократить ручную работу: один интерфейс вместо переключения между приложениями сетей.
  2. Поставить контент «на поток»: тренды → генерация → редактирование → публикация → аналитика.
  3. Автоматизировать привлечение: ключевое слово в комментарии/директе → авто-выдача гайда и ссылки на Telegram-канал.
  4. Видеть, что работает: аналитика по аудитории и по эффективности контента в разрезе направлений.

2. Глоссарий и ключевые сущности

Термин Значение
Направление (Brand / Workspace) Бизнес-направление пользователя. Верхний уровень группировки. Ожидается 4–6 штук.
Аккаунт (Account / Profile) Конкретный аккаунт соцсети (например, Instagram-аккаунт направления «Кофейни»). В Postproxy соответствует «profile».
Profile-группа (Postproxy) Группировка профилей на стороне Postproxy. Используется для изоляции направлений друг от друга.
Пост (Post) Единица контента: текст + медиа + параметры под конкретные сети.
Публикация (Publication) Факт выхода поста в конкретной сети в конкретное время с результатом (успех/ошибка).
Кампания / Воронка (Funnel) Правило авто-ответа: триггер (ключевое слово) → действие (отправить ссылку/гайд).
Медиа-актив (Asset) Файл (фото/видео) в библиотеке, переиспользуемый в разных постах.

3. Технические реалии соцсетей (на 2026 год) — обязательно к учёту

Это не «придирки», а ограничения платформ, которые определяют, что приложение может обещать пользователю. Использование Postproxy снимает часть боли (аудиты, OAuth, токены), но правила самих платформ остаются.

3.1 Telegram — самый простой

  • Полный автопостинг без ограничений через Bot API.
  • Модель «свой бот»: бот создаётся через @BotFather, публикует во все каналы, где он администратор; адресация поста — по chat_id.
  • До 10 фото/видео в одном сообщении (альбом). Поддерживается двусторонняя переписка в личке без 24-часового окна.

3.2 Instagram — публикация только для бизнес-аккаунтов

  • Постинг возможен только для Business/Creator аккаунтов, привязанных к Facebook-странице.
  • Поддерживаются: фото, карусели, Reels. Stories — поддержка нестабильная (Postproxy заявляет Stories, но часть документации Meta помечает Stories/Reels как неподдерживаемые в Content Publishing API; считать «зоной риска» и проверять на этапе интеграции).
  • Лимит публикации: 100 постов за скользящие 24 часа на аккаунт (карусель = 1 пост).
  • Музыку из библиотеки Instagram через API добавить нельзя — звук должен быть вшит в видео.

3.3 Воронка авто-ответа (ключевое слово → личка)

  • Реализуется через Direct Messages API + вебхуки Postproxy (Private Replies — превращение комментария в личку; ig.me-реферралы с атрибуцией кампании).
  • Окна отправки Meta (критично): свободное сообщение можно отправить только в течение 24 часов с последнего входящего от пользователя; private reply на комментарий даёт отдельное окно 7 дней, один ответ на один комментарий. Вне окна — нельзя.
  • Для работы ig.me-реферралов у Instagram-профиля должен быть задан хотя бы один ice breaker.
  • Лимит автоматических личных сообщений — порядка 200 в час на аккаунт; нужна очередь и пейсинг.
  • В Telegram окна нет — после первого сообщения от пользователя бот может отвечать в любой момент.

3.4 TikTok — самый строгий, но аудит уже на стороне Postproxy

  • Полный автомат (Direct Post) обычно требует аудита приложения (2–6 недель). Postproxy этот аудит прошёл, поэтому отдельно проходить его не нужно.
  • Остаются правила платформы: запрет на логотипы/вотермарки/промо-текст поверх контента (нарушение → отключение аккаунта); лимит ~15–25 постов в день на аккаунт; модерация может снять пост уже после публикации.

3.5 Threads — публикация есть, но с нюансами

  • API публикации работает (контейнерная модель). Лимит — 250 постов за 24 часа на аккаунт.
  • Нет нативного планирования (очередь/расписание делаем на своей стороне или через планировщик Postproxy), нет редактирования опубликованного, токены требуют обновления.
  • Лимит текста 500 символов; до 10 изображений; видео до 5 минут.

3.6 Facebook — публикация только на Страницы

  • Постинг идёт на Facebook-страницы (Pages), не в личные профили. Управление через Graph API; нужен Page access token и app review (как у Instagram, это та же экосистема Meta).
  • Поддерживаются: посты в ленту (фото/видео/текст/ссылки), Stories на Страницы, «первый комментарий», управление несколькими Страницами — Postproxy это закрывает.
  • Для воронок: Facebook Messenger поддержан в Direct Messages API Postproxy (личка + private replies), работают те же 24-часовые окна Meta, что и у Instagram.

3.7 X (Twitter) — публикация есть, но дорогой прямой API

  • Через Postproxy публикация абстрагирована: текстовые посты (лимит 280 символов), медиа (до 4 вложений), цепочки-треды (tweet chains).
  • Важный нюанс 2026: прямой X API перешёл на pay-per-use и бесплатного тарифа для новых разработчиков больше нет. Создание поста стоит ≈$0.015, а пост со ссылкой — ≈$0.20 (в ~13 раз дороже). Чтение и часть действий (лайк/подписка/quote) ушли в дорогие/Enterprise-тарифы. Прямое построение на X API дорого и нежелательно — поэтому идём через Postproxy.
  • Для твоего кейса это критично: ты планируешь постить ссылки (на гайды и Telegram-канал). Если когда-либо встанет вопрос прямой интеграции с X в обход провайдера — «налог на ссылки» сделает это дорогим. Через Postproxy этот риск снимается, но стоит держать в голове при выборе тарифа.

3.8 Поиск трендов — честное ограничение

  • Официального «API трендов» у TikTok нет (TikTok Creative Center — это веб, не API). У Instagram есть hashtag search, но он сильно лимитирован (≈30 уникальных хэштегов в неделю на аккаунт).
  • Вывод: модуль трендов в Фазе 2 опирается на (а) ограниченные официальные источники, (б) сторонних поставщиков данных о трендах, либо (в) ручной ввод/полуавтоматический мониторинг. Это надо заложить в ожидания, а не обещать «магический» автопоиск трендов.

Ключевой принцип: слой публикации — абстракция

Весь постинг и работа с личками идут через адаптер публикации, за которым стоит Postproxy. Адаптер изолирован интерфейсом (PublishingProvider), чтобы при необходимости заменить Postproxy на прямые интеграции или другого провайдера без переписывания приложения.

Минимальный контракт адаптера:
- publish(post, targets[]) → per-target results
- schedule(post, datetime, targets[])
- getPublicationStatus(id)
- sendDirectMessage(profile, recipient, content)
- sendPrivateReply(commentId, content)
- listChats(profile) / listMessages(chatId)
- обработка вебхуков: message.received, referral.received, события публикации.

5. Роли и права доступа (RBAC)

Роль Права
Владелец (Owner) Полный доступ: все направления, биллинг, подключение аккаунтов, управление командой, публикация, удаление.
Менеджер направления Полный доступ в рамках назначенных направлений (контент, публикация, аналитика, воронки).
Контентщик (Editor) Создание/редактирование постов и медиа в назначенных направлениях. Публикация — по настройке (сразу или через апрув, см. Фаза 3).
Аналитик (Viewer) Только просмотр аналитики и контента, без публикации.

Требования:
- Доступ выдаётся в разрезе направлений (пользователь видит только свои направления).
- Все действия логируются (audit log): кто, что, когда, в каком направлении.
- Подключение/отключение аккаунтов соцсетей — только Owner.

6. Функциональные модули по фазам

Проект сознательно разбит на фазы, чтобы запустить рабочее ядро быстро и наращивать функциональность.

ФАЗА 1 — Ядро (MVP)

Модуль Содержание
Направления и аккаунты Создание направлений; подключение аккаунтов сетей через Postproxy (OAuth/бот); привязка аккаунтов к направлениям; маппинг направлений на profile-группы Postproxy.
Медиа-библиотека Загрузка фото/видео в Object Storage; теги, поиск, превью; переиспользование одного актива в разных постах; контроль форматов под требования сетей.
Редактор поста Единый текст + адаптация под сети (разные лимиты символов — например, X 280, Threads 500; обрезка, превью под каждую сеть); выбор медиа; пер-сетевые параметры (cover для Reels, first comment для IG/FB, parse_mode и chat_id для TG, треды-цепочки для X/Threads, выбор Facebook-страницы, и т.п.).
Планировщик и календарь Визуальный календарь постов в разрезе направлений и сетей; постановка на дату/время; собственная очередь (для Threads и X — обязательно, т.к. нет нативного планирования); учёт лимитов сетей.
Автопостинг Публикация через Postproxy во все подключённые сети (Instagram, TikTok, Telegram, Threads, Facebook, X); пер-сетевой результат (успех/ошибка с причиной); ретраи; статусы: черновик → запланирован → публикуется → опубликован/ошибка.
Базовая аналитика Подписчики, охваты, базовая вовлечённость по аккаунтам; список постов с метриками; срез по направлениям.
Команда и роли Приглашение пользователей, назначение ролей и направлений, audit log.

Критерий готовности Фазы 1: пользователь может подключить аккаунты всех нужных сетей в рамках направления, загрузить медиа, собрать пост, запланировать его и увидеть факт публикации и базовые метрики.

ФАЗА 2 — Контент-завод и воронки

Модуль Содержание
AI-генерация (kie.ai) Через единый API kie.ai: тексты (посты, идеи, заголовки, хэштеги — чат-модели GPT/Claude), изображения для постов (GPT Image, Nano Banana/Flux и др.), короткие видео для Reels/TikTok (Veo, Kling, Seedance и др.), при необходимости озвучка/музыка (ElevenLabs/Suno). «Переписать под сеть» (адаптация тона/длины под IG/TikTok/Threads/TG/X/FB); генерация на основе брифа направления (тон бренда, ЦА). Подробности в п. 7.4.
Поиск трендов (EnsembleData) Провайдер данных о трендах — EnsembleData: один API покрывает все нужные сети (TikTok, Instagram, Threads, X, Facebook, YouTube) — хэштеги, трендовые посты, музыка, поиск по ключевым словам. Старт на бесплатном/нижнем тарифе, апгрейд по мере роста. Дополняется TikTok Creative Center (бесплатно, вручную) и ручным вводом. Превращение тренда в черновик через AI. Альтернатива/добор нишевых источников — Apify (pay-per-use).
Воронки авто-ответа Конструктор правил: триггер (ключевое слово в комментарии/директе) → действие (отправить текст + ссылку на гайд/Telegram-канал); каналы: Instagram, Facebook Messenger, Telegram; учёт окон Meta (24ч / 7 дней private reply); ice breakers; ig.me-реферралы с атрибуцией; статистика воронки (сколько сработало, конверсия).
Расширенная аналитика Эффективность контента: какие посты/форматы/темы заходят лучше; динамика роста; сравнение направлений; лучшее/худшее по охвату и вовлечённости.

Критерий готовности Фазы 2: пользователь может сгенерировать пост с помощью AI, настроить воронку «ключевое слово → гайд» и увидеть, насколько эффективен контент.

Модуль Содержание

7. Детализация ключевых модулей

7.1 Воронка авто-ответа по ключевому слову (главная «фишка»)

Сценарий (на примере Instagram):
1. В посте призыв: «Напиши ГАЙД в комментарии — пришлю ссылку».
2. Пользователь пишет ключевое слово в комментарий.
3. Postproxy через вебхук (message.received / событие комментария) уведомляет бэкенд.
4. Бэкенд проверяет правило воронки: совпадает ли текст с ключевым словом (с учётом регистра, опечаток, синонимов — настраивается).
5. Бэкенд отправляет private reply на комментарий (открывает 7-дневное окно) и/или личное сообщение с текстом + ссылкой на гайд и Telegram-канал.
6. Действие логируется; обновляется статистика воронки (срабатывания, доставлено, переходы).

Требования к модулю:
- Конструктор правил: направление → аккаунт → пост/все посты → ключевое слово(а) → шаблон ответа (текст + ссылки + опционально вложение).
- Учёт лимитов и окон Meta (очередь, пейсинг ≤ ~200 DM/час/аккаунт, проверка last_inbound_at перед свободной отправкой).
- Антиспам: не отвечать одному пользователю повторно на одно и то же; чёрный список.
- Поддержка Telegram (без окна) и Instagram (с окнами) с разной логикой.
- Метрики: число триггеров, доставленных ответов, переходов по ссылке (через ig.me-реферралы/UTM).

7.2 Редактор и адаптация под сети

  • Единое «тело» поста + переопределения под каждую сеть.
  • Валидация под лимиты до публикации: длина текста (X 280, Threads 500, TG большой, IG/TikTok/Facebook свои), число медиа, форматы, длительность видео.
  • Превью «как будет выглядеть» в каждой выбранной сети.
  • Пер-сетевые поля: Reels cover/cover_url, first comment (IG/Facebook), IG collaborators, Telegram parse_mode (HTML/MarkdownV2) и chat_id, выбор Facebook-страницы, thread-цепочки для Threads и X (Twitter).
  • Предупреждение про TikTok: запрет вотермарок/логотипов поверх контента (валидатор + подсказка).
  • Предупреждение про X: пост со ссылкой при прямой интеграции дороже (через Postproxy не критично, но учитывается в выборе тарифа).

7.3 Планировщик

  • Собственная очередь задач (job queue) с расписанием (cron-подобным).
  • Для Threads и X — публикация строго через свою очередь (нет нативного планирования у платформ).
  • Контроль лимитов по каждой сети/аккаунту (не превышать суточные потолки).
  • Обработка ошибок: ретраи с экспоненциальной задержкой; пометка постоянных ошибок (неверные scope, неподдерживаемое медиа) как non-retryable.
  • Идемпотентность: не публиковать один и тот же пост дважды при ретрае.

7.4 AI-слой на kie.ai

Что это: kie.ai — единый API-агрегатор, дающий через один ключ доступ к множеству моделей: текстовые/чат (GPT, Claude), генерация изображений (GPT Image, Nano Banana/Gemini Flash Image, Flux и др.), генерация видео (Veo, Kling, Seedance, Wan и др.), музыка/озвучка (Suno, ElevenLabs). Это позволяет закрыть весь контент-завод одним поставщиком.

Сценарии использования в приложении:
- Текст: генерация и адаптация постов, идей, заголовков, хэштегов под каждую сеть (с учётом лимитов: X 280, Threads 500 и т.д.).
- Изображения: обложки/визуалы для постов и каруселей по текстовому брифу.
- Видео: короткие ролики для Reels/TikTok/Threads/Shorts (text-to-video / image-to-video).
- Аудио: озвучка и музыкальная подложка для видео (опционально).

Технические требования (важно для разработчика):
- Асинхронная модель: генерация в kie.ai не мгновенная — запрос возвращает taskId, статус опрашивается через polling или приходит на callback URL. Приложение должно обрабатывать генерацию как фоновую задачу (очередь + статусы: «генерируется → готово → ошибка»), а не как синхронный ответ.
- Хранение результатов: сгенерированные медиа на стороне kie.ai хранятся ограниченное время (например, видео — порядка 14 дней). Приложение должно сразу скачивать результат в своё Object Storage и работать уже со своей копией.
- Безопасность ключа: ключ kie.ai — серверный секрет, никогда не на фронтенде/в репозитории.
- Вотермарка ↔ TikTok (критично): часть видео-моделей умеет ставить вотермарку. Для TikTok вотермарка запрещена (см. п. 3.4) — при генерации видео под TikTok вотермарку нужно отключать (пустая строка вотермарки / соответствующий параметр модели). Заложить это правилом в пайплайн.
- Выбор модели и стоимость: kie.ai дешевле официальных API (обычно на 30–50%, иногда до 80%), цены зависят от модели и меняются — сверять по их pricing-странице; в приложении предусмотреть выбор модели под задачу (быстрая/дешёвая vs качественная).
- Абстракция провайдера: AI-слой так же изолировать интерфейсом (AIProvider), чтобы при необходимости заменить или дополнить kie.ai другим поставщиком.

Зависимости и риски провайдера:
- Тарификация по объёму постов и числу profile-групп (есть бесплатный тариф). Точные цены сверить на postproxy.dev перед стартом — они меняются.
- Глубина аудитории-аналитики ограничена тем, что отдают сами платформы (например, ряд метрик Instagram Insights был урезан Meta — profile_views, website_clicks и др.). Закладывать только реально доступные метрики.

10. Нефункциональные требования

  • Безопасность: секреты в защищённом хранилище (Vault/секрет-менеджер), не в репозитории; шифрование данных; строгий доступ по ролям.
  • Приватность/GDPR: Postproxy хранит данные на EU-инфраструктуре и GDPR-совместим; со своей стороны — политика хранения, право на удаление, минимизация ПДн. Не помещать персональные данные в URL/query-параметры.
  • Надёжность: очереди, ретраи, идемпотентность, пер-сетевой учёт частичных успехов (один сбой не должен ронять публикацию в остальные сети).
  • Лимиты: соблюдение суточных потолков каждой сети и лимита DM/час; пейсинг.
  • Наблюдаемость: логи, статусы публикаций, дашборд ошибок «почему не вышло».
  • Масштаб (ориентир): 4–6 направлений × ~2–4 аккаунта ≈ 8–24 аккаунта; десятки–сотни постов в неделю + фоновые AI-генерации. Архитектура должна спокойно держать рост до 10+ направлений.
  • Локализация: интерфейс на русском (минимум), задел под мультиязычность.

12. Бюджет ($100–300/мес) и контроль расходов

Целевой бюджет на внешние сервисы — $100–300 в месяц. Он реалистичен, но решающий фактор — объём генерации видео (самая дорогая статья). Ниже ориентир по статьям (точные цифры зависят от объёмов и тарифов — сверять на сайтах сервисов).

Статья Ориентир/мес Комментарий
Postproxy (публикация + личка) ~$30–80 Тариф по объёму постов и числу profile-групп (4–6 групп). Есть бесплатный старт.
kie.ai — текст ~$2–10 Чат-модели для текстов/идей/хэштегов — копейки.
kie.ai — изображения ~$5–30 ~$0.02–0.10 за картинку; 150–300 шт/мес.
kie.ai — видео ~$40–120 Главная переменная. Быстрые модели (Veo 3 Fast ≈$0.40/8с, Wan/Seedance ≈$0.20–0.40, Kling ≈$0.07/с) при 720p. С учётом 2–3 дублей — ~$0.6–1.2 за готовый ролик → ~60–180 роликов/мес.

Требования к приложению для удержания бюджета (обязательно):
- Учёт расходов: считать траты на AI/публикацию по направлению и за месяц, показывать на дашборде.
- Лимиты и алерты: настраиваемый месячный потолок (общий и/или по направлению) с предупреждением и мягкой остановкой генерации при достижении.
- Дефолт — дешёвые модели: для соцсетей использовать быстрые модели и 720p (этого достаточно, сети всё равно сжимают видео); премиум-модели (Veo Quality и т.п.) — только для «героического» контента по явному выбору.
- Экономия на переиспользовании: медиа-библиотека и повторное использование контента снижают число повторных генераций.
- Отключение вотермарки для TikTok (см. п. 3.4, 7.4) — заодно избегаем перегенераций из-за отклонённых постов.

13. Метрики успеха (KPI приложения)

  • Время на подготовку и публикацию поста (цель: ↓ в разы против ручного режима).
  • Доля успешных публикаций без ручного вмешательства (цель: 95%+ там, где API позволяет).
  • Число лидов из воронок авто-ответа (срабатывания → переходы → подписки в TG).
  • Рост аудитории и вовлечённости по направлениям (трекинг динамики).
  • Регулярность постинга (соблюдение контент-плана).

14. Предлагаемый стек (на усмотрение разработчика)

  • Фронтенд: React/Next.js + TypeScript; календарь (drag-and-drop), редактор с превью.
  • Бэкенд: Node.js (NestJS) или Python (FastAPI); REST или GraphQL.
  • БД: PostgreSQL; кэш — Redis.
  • Очереди/планировщик: BullMQ/Redis или Celery; cron-планировщик публикаций.
  • Хранилище медиа: S3-совместимое Object Storage + CDN (медиа должно быть по публично доступному URL — этого требуют API Meta/Threads).
  • Публикация/личка: Postproxy (Posts, Direct Messages, Comments, Webhooks). Есть SDK (Node/PHP/Go) и MCP-сервер.
  • AI: kie.ai (единый API): текст, изображения, видео, аудио. Асинхронная генерация — нужны очередь фоновых задач и обработка callback-ов; результаты сразу складывать в Object Storage.
  • Тренды: EnsembleData (TikTok/IG/Threads/X/FB/YouTube); опц. Apify для нишевых источников.
  • Контроль расходов: учёт трат по направлению/месяцу + лимиты и алерты (раздел 12).
  • Секреты: секрет-менеджер (Vault/cloud secrets).

16. Зафиксированные решения и оставшиеся вопросы

Решено:
1. Аккаунтов на направление: ~2–4 (итого ≈ 8–24 аккаунта).
2. Сети в составе: Instagram, TikTok, Telegram, Threads, Facebook, X (Twitter). Остальные из списка Postproxy (YouTube, LinkedIn, Pinterest, Bluesky, Google Business) — на будущее, добавляются легко через тот же провайдер.
3. AI-генерация: kie.ai (текст + изображения + видео + аудио через один API).
4. Тренды: провайдер EnsembleData (покрывает все нужные сети); старт на бесплатном/нижнем тарифе.
5. Апрув-воронка: не нужна (исключена из ТЗ).
6. Бюджет: $100–300/мес на внешние сервисы; обязателен учёт расходов и лимиты в приложении (раздел 12).
7. Видео-генерация: входит в Фазу 2 (не откладывается). Для удержания бюджета — быстрые модели и 720p по умолчанию.

Всё ключевое зафиксировано. Перед стартом разработчику остаётся лишь подтвердить ориентир месячных объёмов (постов и генераций) для точного расчёта тарифов Postproxy / kie.ai / EnsembleData и оценки сроков.

Документ описывает концепцию и требования. Реализацию, оценку сроков и архитектурные детали финализирует разработчик после ревью.
